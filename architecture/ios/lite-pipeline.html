Takes the approved spec and writes production-quality Swift code following MVVM-C + Clean Architecture. Not a "dump code and hope" agent ‚Äî it follows a structured 4-phase process: plan the architecture first, then build each layer, then wire everything together.

4-Phase Process
Phase 1 ‚Äî Architect
Principal
Plan Before Code
Reads the spec. Researches Swift/iOS best practices via web search. Decides: protocol contracts between layers, dependency graph, file structure, error types, coordinator navigation plan. Writes an ADR (Architecture Decision Record) explaining the structural choices. No code until this is done.
Phase 2 ‚Äî Domain Layer
Senior
Domain Lead
Pure Swift entities, use case protocols, repository protocols. Zero framework imports ‚Äî this layer is the business logic with no iOS dependencies.
Phase 3 ‚Äî Parallel
Senior
Data Lead
DTOs, mappers, API clients, repository implementations. Implements the protocols defined in Domain.
Senior
Presentation Lead
ViewModels, SwiftUI Views, Coordinators. Consumes use cases from Domain. Implements design tokens from the spec.
Phase 4 ‚Äî Integration
Staff
Wire + Audit
DI container assembly, coordinator setup, protocol conformance audit. Verifies Domain has zero framework imports, Presentation doesn't touch Data directly. Runs SwiftLint + SwiftFormat. Fixes what it can, flags what it can't.
?

Why 4 phases and not just "write code"?
Because AI writes better code when it thinks in layers. The Architect plans structure. Domain stays pure. Data and Presentation build in parallel (they don't depend on each other). Integration catches wiring mistakes. Same reason good teams work this way.

Lite vs Full Pipeline
Lite (this one)
  • 4 stages, zero configuration
  • No bootstrap ‚Äî just start
  • No pipeline memory ‚Äî each run is fresh
  • No configurable stages ‚Äî everything always runs
  • No task routing ‚Äî one flow for all tasks
  • No deploy stage ‚Äî you handle that
  • Built-in lint (inside Coder) instead of separate agent
  • Perfect for: side projects, learning, MVPs, hackathons
Full Pipeline
  • 10+ agents, bootstrap configures everything
  • Bootstrap detects context, discovers MCPs, sets preferences
  • Pipeline Memory ‚Äî learns from every run, avoids past mistakes
  • Test, Lint, Build, Deploy all independently configurable
  • 9 task routing paths ‚Äî each task type takes the shortest path
  • Deploy stage for TestFlight / App Store
  • Separate Test + Lint agents run in parallel
  • Built for: production apps, growing projects, teams
‚Üë

When to upgrade: When you find yourself wanting to skip stages for certain tasks, when you want the pipeline to remember past decisions, or when you add deploy targets. The full pipeline is a superset ‚Äî everything here exists there too.

Design Principles

You always have the final say. Two explicit human checkpoints: you approve the spec before coding starts, and you review the PR before it merges. The pipeline never commits to something irreversible on its own.

Standards are built in, not optional. The Coder follows MVVM-C + Clean Architecture with protocol-driven DI by default. Domain layer has zero framework imports. Presentation never touches Data. SwiftLint enforces this automatically. You get clean code without configuring anything.

Planning before coding. The Architect sub-phase researches and plans the full structure before a single line of code is written. This is what separates good AI output from spaghetti ‚Äî the same way a senior dev would sketch the architecture before coding.

Parallel where it makes sense. Data Lead and Presentation Lead work simultaneously because they don't depend on each other (both implement Domain protocols). This is faster and mirrors how real iOS teams work.

Self-healing up to a point. Compile errors loop back automatically. Lint violations auto-fix. But if the spec is genuinely unclear, the pipeline stops and asks you instead of guessing. It knows when to ask for help.
