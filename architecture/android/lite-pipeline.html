Stage 2
‚öôÔ∏è Coder ‚Äî Writes Clean, Structured Kotlin
Takes the approved spec and writes production-quality Kotlin code following MVVM + Clean Architecture with Jetpack Compose. Not a "dump code and hope" agent ‚Äî it follows a structured 4-phase process: plan the architecture first, then build each layer, then wire everything with Hilt.

Architecture: MVVM + Clean Architecture. Hilt for DI. Kotlin Coroutines + Flow for async. Navigation Compose for routing. Room for local persistence. Retrofit/Ktor for networking. Unidirectional data flow (UDF) in ViewModels.
4-Phase Process
Phase 1 ‚Äî Architect
Principal
Plan Before Code
Reads the spec. Researches Kotlin/Android best practices via web search. Decides: module structure (feature modules if needed), interface contracts between layers, Hilt module plan, Navigation graph, error handling strategy. Writes an ADR explaining structural choices. No code until this is done.
Phase 2 ‚Äî Domain Layer
Senior
Domain Lead
Pure Kotlin data classes, use case classes, repository interfaces. Zero Android imports ‚Äî this layer is business logic only. Uses Kotlin Coroutines for async contracts (suspend functions, Flow return types).
Phase 3 ‚Äî Parallel
Senior
Data Lead
DTOs, mappers, Retrofit/Ktor API services, Room DAOs + entities, repository implementations. Provides Hilt @Binds for repository interfaces. Implements offline-first with Room where applicable.
Senior
Presentation Lead
ViewModels (with UiState sealed classes), Jetpack Compose screens + components, Navigation Compose routes. Consumes use cases from Domain. Implements M3 design tokens from the spec.
Phase 4 ‚Äî Integration
Staff
Wire + Audit
Hilt module assembly (@Module, @Provides, @Binds), NavHost setup, ViewModel injection verification. Verifies Domain has zero Android imports, Presentation doesn't reference Data layer directly. Runs ktlint + detekt. Fixes what it can, flags what it can't.
?

Why 4 phases and not just "write code"?
Because AI writes better code when it thinks in layers. The Architect plans module structure. Domain stays pure Kotlin. Data and Presentation build in parallel (they both implement Domain interfaces). Integration catches Hilt wiring mistakes. Same reason Google's architecture guide recommends this separation.

Android Standards ‚Äî What the Coder Enforces
Architecture
MVVM + Clean Architecture. Domain layer is pure Kotlin (no Android imports). Data layer implements Domain interfaces. Presentation consumes via ViewModel + StateFlow.
DI & Navigation
Hilt for dependency injection (@HiltViewModel, @Inject, @Module). Navigation Compose with type-safe routes. Single Activity architecture.
Async & State
Kotlin Coroutines + Flow. ViewModels expose UiState via StateFlow. Sealed classes for UI states (Loading, Success, Error). Unidirectional data flow.
Lint & Format
ktlint for code formatting (auto-fix enabled). detekt for static analysis. Layer boundary checks (Domain ‚â† Android imports). All enforced automatically in Integration phase.

Same pipeline, different standards. The 4-stage flow is identical. The 4-phase Coder process is identical. What changes is the language, frameworks, build tools, and design guidelines. The architecture principles (clean layers, pure domain, parallel work) are universal.

Design Principles

You always have the final say. Two explicit human checkpoints: you approve the spec before coding starts, and you review the PR before it merges. The pipeline never commits to something irreversible on its own.

Standards are built in, not optional. The Coder follows MVVM + Clean Architecture with Hilt DI by default. Domain layer has zero Android imports. Presentation never touches Data directly. ktlint and detekt enforce this automatically. You get clean code without configuring anything.

Planning before coding. The Architect sub-phase researches and plans the full module structure before a single line of code is written. This is what separates good AI output from spaghetti ‚Äî the same way a senior dev would design the navigation graph and Hilt modules before coding.

Parallel where it makes sense. Data Lead and Presentation Lead work simultaneously because they don't depend on each other (both implement Domain interfaces). This is faster and mirrors how real Android teams work.

Self-healing up to a point. Compile errors loop back automatically. Lint violations auto-fix via ktlint --format. But if the spec is genuinely unclear, the pipeline stops and asks you instead of guessing. It knows when to ask for help.
